# Loop Task

## 需求澄清
- 现状：抽牌堆为空时，直接判平局（`GameManager.drawCards` 里设置 `isDraw=true`）。
- 目标：改为“保留弃牌堆顶牌，其余弃牌重新洗回抽牌堆”，让游戏继续。
- 风险：在“随机出牌”的模拟对局中可能出现死循环/无限重洗。
- 需要：加入保险机制，避免测试或对局无限循环。

## 约束与范围
- 影响模块：
  - `server/src/game/GameManager.ts`（抽牌逻辑）
  - `server/src/game/Simulation.test.ts`（随机对局测试）
  - `shared/src/types/game.ts`（若平局规则变化，可考虑是否仍保留 `isDraw` 字段）
  - `client/src/pages/Game.tsx`（如平局规则仍存在，提示逻辑是否保留）

## 需求分解
1. **规则定义**
   - 抽牌堆为空时：
     - 如果弃牌堆有 2 张及以上：保留顶牌，其余洗回抽牌堆继续。
     - 如果弃牌堆仅 1 张：无法重洗（没有可回收牌），此时再进入保险机制（可能判平局或触发上限）。

2. **保险机制设计（防死循环）**
   - 方案 A（推荐）：
     - 为每局记录“重洗次数”或“回收次数”，超过阈值则判平局。
   - 方案 B：
     - 检测连续 N 次“抽牌堆为空且无法产生可出牌动作”，触发平局。
   - 建议：**使用重洗次数阈值 + 回合数上限双保险**（测试环境更稳定）。

3. **服务端实现步骤**
   1) 在 `GameState` 加入 `reshuffleCount` 或 `recycleCount` 字段（如需持久化）。
   2) 修改 `drawCards`：
      - 抽牌堆空时，保留弃牌堆顶牌，其余洗回抽牌堆。
      - 若无法回收（弃牌堆 <= 1），触发保险机制。
   3) 保险机制触发时：
      - 设置 `phase='finished'` + `isDraw=true`。
      - 清理 `pendingPenalty` / `challengeState`。

4. **测试更新**
   - `Simulation.test.ts`：
     - 增加断言：重洗次数不会无限增长。
     - 允许平局，但必须满足“重洗阈值/回合上限触发”。
   - 增加单测：
     - 抽牌堆为空且弃牌堆多张时 → 继续游戏。
     - 抽牌堆为空且弃牌堆仅 1 张 → 触发保险机制。

5. **客户端展示**
   - 若平局仍可能发生（保险机制触发）：保留 `isDraw` 展示逻辑。
   - 否则可考虑改为“异常终止提示”。

## 执行顺序
1. 定义保险机制策略（阈值、触发条件）。
2. 修改 `GameState` 结构与默认值。
3. 更新 `drawCards` 重洗逻辑。
4. 更新平局/结束逻辑。
5. 更新/补充测试用例。
6. 运行 `npm run test --workspace=server` 验证。

## 交付物
- 抽牌堆重洗逻辑完成，平局仅在保险机制触发时出现。
- 模拟对局测试稳定通过。
- 规则与测试文档更新（如需要）。
