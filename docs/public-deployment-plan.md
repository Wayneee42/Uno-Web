# Uno 公网联机简化方案

## 1. 目标

这份文档讨论的不是正式商用上线，而是一个更现实的目标：

把当前项目放到公网上，让朋友不在同一个局域网里也能直接通过链接联机。

所以这里的重点是：

- 方案尽量简单
- 成本尽量低
- 部署尽量省事
- 对当前项目改动尽量可控

## 2. 可以不用自己买云服务器吗

可以。

对于这个项目来说，更适合的路线其实不是一开始就自己买云服务器手动部署，而是：

- 前端放在静态托管平台，例如 Vercel 或 Netlify
- 后端放在支持 Node 服务和 WebSocket 的平台，例如 Render Web Service 或 Railway

这套组合比“自己买一台服务器再手工配 Nginx 和证书”更轻，尤其适合下面这种场景：

- 只是和朋友联机
- 同时在线人数不多
- 想先尽快跑起来
- 不想先投入太多运维精力

## 3. 推荐方案

### 3.1 前端

前端推荐放在：

- Vercel
- Netlify

原因：

- 都很适合 Vite 打包后的静态站点
- 自动构建和自动部署体验比较好
- HTTPS、CDN、域名绑定这些基础能力基本现成
- 不需要自己维护静态文件服务器

### 3.2 后端

后端推荐放在：

- Render Web Service
- Railway

原因：

- 都支持 Node 服务
- 都比自己买服务器再配环境更省事
- 更适合当前这种小规模实时联机项目
- 足够承载 Socket.IO 这类实时通信场景的早期版本

### 3.3 整体结构

推荐结构是：

- `client` 部署到 Vercel 或 Netlify
- `server` 部署到 Render 或 Railway
- `shared` 继续作为 monorepo 里的共享类型包参与构建
- 前端通过环境变量连接后端公网地址

也就是说，最终访问路径会变成：

- 用户打开前端站点，例如 `https://uno-your-app.vercel.app`
- 前端再连接后端，例如 `https://uno-server.onrender.com`

## 4. 这种方案相对自购服务器的优缺点

### 4.1 优点

- 不用先折腾 Linux 服务器
- 不用自己配 Nginx、证书、进程守护这些基础设施
- 前端部署非常快
- 后端上线门槛低
- 适合先验证“公网能不能顺畅玩”

### 4.2 缺点

- 免费套餐通常有休眠、冷启动或资源限制
- WebSocket 长连接稳定性会受平台套餐和配置影响
- 可控性没有自建服务器高
- 如果以后人数变多，可能还是要迁移

所以这套方案很适合“朋友联机版”，但不一定适合长期大规模公开运营。

## 5. 如果采用这种方案，最少需要补哪些东西

### 5.1 前端连接配置

这是最先要处理的。

当前项目如果还是偏向本地开发方式推导服务地址，那么上到 Vercel 或 Netlify 后就容易出问题。

建议补的内容：

- 让前端明确读取 `VITE_SERVER_URL`
- 生产环境下不要再依赖 `window.location.hostname + 3001` 这类本地逻辑
- 本地、局域网、公网三种场景的连接方式要能共存

目标是：

- 本地开发时照常连接本地服务
- 部署后前端明确连接 Render 或 Railway 上的后端地址

### 5.2 后端跨域配置

既然前端和后端很可能分开部署，就必须明确处理跨域。

建议补的内容：

- `CLIENT_ORIGIN` 支持前端实际域名
- 如果要同时支持本地调试和线上前端，需要支持多个 origin
- 在部署文档里写清楚前后端地址如何配套

否则很容易出现：

- 页面能打开
- 但 Socket.IO 连不上
- 或者浏览器直接 CORS 报错

### 5.3 WebSocket 可用性验证

因为项目核心是实时房间联机，所以不是“页面能打开”就算部署成功。

必须确认：

- Socket.IO 能在平台环境下正常连接
- 房间创建、加入、出牌这些流程能走通
- 断线重连逻辑在线上环境仍然可用

这个部分尤其重要，因为静态站点部署很简单，但实时后端才是核心风险点。

### 5.4 邀请链接体验

既然是给朋友玩，最实际的优化还是：

- 复制房间号
- 复制邀请链接
- 打开链接后自动带入房间号

如果前端已经有一部分这类能力，建议在文档里把它明确列成上线前必须验证的一项。

### 5.5 服务端最基本的防呆

即使不是正式商用，也建议补最基本的健壮性：

- 校验关键 socket payload
- 房间空了以后及时清理
- 玩家超时未重连时能回收状态
- 异常情况下不要把整个房间状态搞乱

这部分不是为了“高强度安全防护”，而是为了避免线上偶发问题直接影响朋友联机体验。

## 6. 平台选择建议

### 6.1 Vercel vs Netlify

如果只看当前项目，二者都可以。

更偏向 Vercel 的理由：

- React/Vite 项目接入普遍更顺手
- 前端部署体验通常更直接
- 文档和生态更常见

更偏向 Netlify 的理由：

- 也很适合静态站点
- 配置方式清晰
- 对简单站点托管同样够用

结论：

- 如果你没有明显偏好，前端优先试 Vercel
- 如果你已经更熟悉 Netlify，也完全可以直接用 Netlify

### 6.2 Render vs Railway

如果只看这个项目的后端托管，二者也都可行。

更偏向 Render 的理由：

- 对“一个长期运行的 Node Web Service”这个模型更直观
- 部署方式比较接近传统后端服务认知
- 文档和案例相对多一些

更偏向 Railway 的理由：

- 上手很快
- 配置界面友好
- 做小型项目体验不错

结论：

- 如果你想要一个更接近常规后端服务的托管方式，优先试 Render
- 如果你更看重快速跑起来和界面体验，Railway 也很合适

## 7. 推荐的实际组合

对这个项目，我建议优先按下面这个组合尝试：

- 前端：Vercel
- 后端：Render Web Service

理由：

- 前端静态托管最省心
- 后端作为长期运行的 Node 服务比较好理解
- 对“朋友联机版”来说，这个组合已经足够

如果你更喜欢 Railway，也可以用：

- 前端：Vercel
- 后端：Railway

本质上前端选 Vercel/Netlify、后端选 Render/Railway 都能成立，关键不是平台名称，而是把连接配置和联机流程跑通。

## 8. 推荐的开发和部署流程

### 第一步：先把前端服务地址配置整理好

先做代码层的准备，不要先急着部署。

要做的事：

- 确认前端统一通过 `VITE_SERVER_URL` 连接后端
- 明确开发环境和线上环境的取值方式
- 确保本地不受影响，线上可以直接替换后端地址

完成标准：

- 本地开发能跑
- 手动指定一个线上地址时，前端也能正确连接

### 第二步：整理后端环境变量

要做的事：

- 明确 `PORT`
- 明确 `CLIENT_ORIGIN`
- 明确日志等级等基础变量
- 写清楚 Render 或 Railway 里要填哪些环境变量

完成标准：

- 后端部署到托管平台后无需修改代码，只靠环境变量就能运行

### 第三步：先部署后端

建议先上后端，因为前端最终要指向它。

要做的事：

- 在 Render 或 Railway 上创建 Node 服务
- 配置构建命令和启动命令
- 配置环境变量
- 拿到公网服务地址
- 验证健康检查和 Socket 连接

完成标准：

- 后端地址可访问
- 房间相关接口和 Socket.IO 可以连通

### 第四步：再部署前端

要做的事：

- 在 Vercel 或 Netlify 上连接仓库
- 设置前端构建目录和构建命令
- 配置 `VITE_SERVER_URL` 指向后端地址
- 部署完成后验证页面可正常加载

完成标准：

- 页面能打开
- 可以成功连接后端
- 不出现跨域和协议错误

### 第五步：做真实联机测试

这一步不能省。

至少要验证：

- 两个不同网络环境的人能进入同一房间
- 创建房间、加入房间、开始游戏、出牌、结束流程都正常
- 断线后能在预期时间内重连
- 一方刷新页面后，session 恢复逻辑正常
- 邀请链接流程顺畅

完成标准：

- 不只是“能打开”，而是真的能完整玩一局

### 第六步：整理最小维护说明

这部分不用很正式，但最好写下来。

至少写清楚：

- 前端部署在哪个平台
- 后端部署在哪个平台
- 环境变量填什么
- 改代码后怎么重新部署
- 出问题时先检查什么

## 9. 这种方案下的实际前提清单

如果你决定走“Vercel/Netlify + Render/Railway”路线，至少要准备：

- 一个 Git 仓库
- 一个前端托管账号
- 一个后端托管账号
- 一套清晰的环境变量
- 一个最终可公开访问的前端 URL
- 一个最终可公开访问的后端 URL

如果以后想更正式一点，再考虑：

- 自定义域名
- 更稳定的套餐
- 更完善的日志和监控

## 10. 最适合当前项目的结论

如果目标只是让朋友不在同一局域网也能联机，这个项目完全可以不先买云服务器。

更合适的第一步是：

- 前端用 Vercel 或 Netlify 托管
- 后端用 Render Web Service 或 Railway 托管
- 优先把 `VITE_SERVER_URL`、`CLIENT_ORIGIN`、Socket.IO 联机流程跑通

这条路线的优点是：

- 上手快
- 成本低
- 运维负担小
- 足够验证这个项目在公网环境下是否好用

如果后面发现：

- 免费套餐限制明显
- 冷启动影响体验
- 想要更稳定的长连接
- 想要更高可控性

再迁移到自购云服务器也不迟。

## 11. 下一步最值得先做的两件事

如果继续推进，我建议下一步优先做这两件事：

1. 明确并重构前端连接后端的配置方式，让线上环境只依赖 `VITE_SERVER_URL`。
2. 写一份针对 `Vercel + Render` 或 `Vercel + Railway` 的具体部署步骤文档。
