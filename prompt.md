# 联机卡牌游戏 "Uno-Web" 项目开发指南

## 1. 项目概述
你现在是一位资深的全栈游戏开发工程师。我们将从零开发一款基于 Web 的联机卡牌游戏（规则参照现代版 UNO 优诺）。
作为我的 AI 助手，你需要提供清晰、模块化、可运行的代码，并严格遵循下述技术栈、架构规范和游戏规则。

## 2. 技术栈
- **前端**: React 18, Vite, TypeScript, Tailwind CSS, Framer Motion, Socket.io-client
- **后端**: Node.js, Express, TypeScript, Socket.io
- **包管理器**: npm

## 3. 核心架构原则：权威服务器 (Authoritative Server)
- **前端为“哑终端”**: 前端只负责渲染游戏状态（State）和发送玩家意图（Action, 如 "PLAY_CARD", "DRAW_CARD"）。
- **后端掌握绝对真理**: 后端维护完整的 `GameState`，处理所有的逻辑校验。绝不信任前端传来的计算结果。
- **状态同步**: 后端每次状态变更后，将当前玩家可见的 `GameState` 广播给该房间内的前端（注意隐藏其他玩家的手牌详情，只发送手牌数量）。

## 5. Uno游戏规则

### 一、 游戏目标与基本构成
* **核心目标**：成为第一个**出光手中所有牌**的玩家。
* **获胜方式**：现代直接胜负制（率先出完手中所有牌即视为赢得该局）。
* **牌组构成（现代版 112 张）**：
  * **数字牌 (80张)**：红、黄、绿、蓝四色。每种颜色包含 2套“0-9”。
  * **功能牌 (24张)**：红、黄、蓝、绿四色。每色各包含 2张抽二(Draw2)、2张反转(Reverse)、2张跳过(Skip)。
  * **万能牌 (8张)**：4张万用牌(Wild) 和 4张王牌抽四(Wild Draw4)。

### 二、 游戏准备（Setup）
1. **确定庄家**：每人随机抽一张牌，**数字最高者为庄家**（功能牌/万能牌计为0分）。
2. **发牌**：向每位玩家分发 **7张** 牌。
3. **起始弃牌**：剩余牌堆面朝下放置形成“抽牌堆”(Draw Pile)，翻开第一张牌放在旁边形成“弃牌堆”(Discard Pile)。
4. **特殊首张处理**：
   * 若是**抽二、反转、跳过**：放入弃牌堆同样作为第一张牌，并按照该牌颜色开始游戏。
   * 若是**万用牌、王牌抽四、洗牌卡**：放回抽牌堆，重新抽取直至卡牌带有基础颜色。

### 三、 游戏玩法与规则
1. **出牌规则**：庄家决定初始出牌方向（顺时针或者逆时针）。每位玩家打出的牌，必须与弃牌堆顶部的牌在**颜色、数字或符号**上保持一致（万能牌除外）。
2. **抽牌机制**：
   * **无牌可出**：必须从抽牌堆**抽一张牌**。
   * **抽后立即出牌**：若抽到的牌符合出牌规则，玩家可选择立即打出。否则回合结束，轮到下家。
   * **放弃出牌**：玩家即使有可匹配的牌也可以选择不出，但此时必须抽一张牌；抽牌后**只能**选择打出刚抽到的牌，不能打出手里原有的牌。
3. **喊出“UNO”**：当玩家打出倒数第二张牌，手里**仅剩一张牌**时，必须触发“喊 UNO”状态（前端需有专门的交互）。

### 四、 功能牌详细规则
* **抽二牌 (+2)**：下家强制抽2张牌，并失去本轮出牌权。
* **反转牌 (Reverse)**：改变出牌方向。
* **跳过牌 (Skip)**：下家失去本轮出牌权。
* **万用牌 (Wild)**：可无视颜色打出，并指定接下来的匹配颜色。
* **王牌抽四 (+4)**：可无视颜色打出，指定接下来的匹配颜色，且下家抽4张牌并失去出牌权。
  * **+4 挑战机制**：下家若怀疑出牌者手里**还有与上一张牌颜色匹配的牌**，可发起挑战。若有，则下家挑战成功；若无，则下家挑战失败。
    * **挑战成功**：出牌者（上家）因违规，需自己抽4张牌，下家免除惩罚。
    * **挑战失败**：挑战者（下家）除原有4张外，需额外罚抽2张（共抽6张），并失去出牌权。

### 五、 特殊规则，依然须遵守。
* **叠加机制（Stacking）**：允许叠加惩罚。若面临被 +2/+4 惩罚，玩家可通过打出 +2 或 +4 （无需对应，+2 或 +4 均可），将累积的罚牌数转嫁给下一位玩家。

## 6. 开发流程 (分阶段实施)

**特别提示**：本项目需同时完美支持 **3人** 和 **4人** 局。所有涉及索引计算的逻辑，必须使用 `players.length` 动态计算，严禁硬编码为 4。

- **第一阶段：基础架构与类型定义 (Monorepo Setup)**
  - 初始化项目结构：`/client` (Vite+React) 和 `/server` (Node+Express)。
  - **关键步骤**：在根目录建立 `/shared` 文件夹（或在两端复制），严格定义 `Card`, `Player`, `GameState` 接口。
  - 配置 Socket.io：实现基础的 `join_room` 事件，确保服务器能区分不同房间的 socket 连接。

- **第二阶段：核心逻辑引擎 (Backend Only)**
  - **2.1 牌堆系统**：根据规则生成 **112张牌**（含80张数字牌）。实现洗牌算法 (Fisher-Yates)。
  - **2.2 动态回合控制器**：
    - 实现 `NextPlayer` 算法：`nextIndex = (currentIndex + direction + totalPlayers) % totalPlayers`。
    - 确保 `Reverse` 牌仅修改 `direction` (1 或 -1)，不依赖固定人数。
  - **2.3 状态校验核心**：
    - 实现 `canPlayCard(handCard, topCard)` 函数。
    - 实现 **混合叠加机制 (Stacking)** 逻辑：增加 `pendingPenalty` 状态。当 `pendingPenalty > 0` 时，玩家只能出 +2 或 +4，否则必须吃下罚牌。

- **第三阶段：房间与大厅系统**
  - **3.1 动态房间管理**：
    - 房主创建房间。
    - 玩家加入（限制人数：最小3人，最大4人）。
    - 实现“全员准备 (Ready)”逻辑：只有当房间内 `readyCount === players.length` 时，房主才可开始游戏。
  - **3.2 游戏初始化**：
    - 服务器处理开局：确定庄家 -> 发7张牌 -> 翻首张牌（处理首张是功能牌的特殊情况）-> 广播初始 `GameState`。

- **第四阶段：前端自适应 UI 开发**
  - **4.1 资源准备**：使用 Tailwind CSS 编写卡牌样式（颜色/数字/图标）。
  - **4.2 动态座位布局 (关键)**：
    - 编写 `TableLayout` 组件。
    - **自身视角固定在底部**。
    - 计算其他玩家位置：根据 `(myIndex - playerIndex + totalPlayers) % totalPlayers` 计算相对位置（Left, Top, Right）。
    - **3人局适配**：UI 应自动隐藏“Right”位置，或调整 Top 位置居中。
  - **4.3 交互实现**：
    - 拖拽或点击出牌 -> Socket 发送 `PLAY_CARD`。
    - 错误反馈：若服务器驳回出牌（如违反规则），前端需弹出提示或震动。

- **第五阶段：复杂交互与动画 (Polishing)**
  - **5.1 挑战机制 (+4 Challenge) UI**：
    - 当有人出 +4 时，下家界面弹出模态框：[接受惩罚] 或 [发起挑战]。
    - 处理挑战结果的动画展示（展示上家手牌）。
  - **5.2 喊 UNO 交互**：
    - 添加“UNO”按钮。
    - 逻辑判定：当手牌剩2张且打出1张时，必须在打出前或打出后2秒内点击（根据具体实现难度调整），否则服务器标记为未喊。
  - **5.3 动画系统 (Framer Motion)**：
    - 实现 `AnimatePresence` 处理发牌、抽牌、出牌的位移路径。
    - 确保动画时长与后端状态更新同步，避免状态跳变。
